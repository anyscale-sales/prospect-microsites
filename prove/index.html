The Opportunity

Primary Use Case

Industry & Use Cases: Prove’s solutions are used across banking, fintech lending, cryptocurrency, payments, healthcare, insurance, e-commerce marketplaces, online gaming, and more. Typical use cases include digital account opening (boosting auto-approvals while screening fraud), customer login authentication (invisibly verifying returning users via phone possession and behavior), and transaction risk checks (e.g. scoring a payment or withdrawal for SIM swap risk in milliseconds). The platform also helps businesses meet regulatory requirements like KYC/AML – e.g. Prove’s identity verification can incorporate sanctions/PEP screening as part of the workflow. Overall, Prove positions itself as “the modern way of proving identity”, emphasizing high accuracy (reducing false declines and fraud) with low friction, enabled by its proprietary blend of machine learning and cryptographic phone-based authentication. AI/ML Initiatives & Use Cases (facts, recent-first) Dec 2025 – ProveX Trust Exchange Launch: Prove introduced ProveX℠, the first “digital trust exchange,” which uses AI/ML-vetted identity tokens to let companies instantly share verified data with partners. For example, a bank using ProveX can confirm a user’s identity once and then fetch that user’s verified credit file or payment credentials from a partner without new checks. ProveX is built on the tokenized identity graph and ensures each data request is tied to a real, authenticated user (not a bot or imposter) via Prove’s ML-driven identity trust checks and cryptographic keys. This initiative extends Prove’s ML-based trust decisions beyond a single enterprise to an entire network, enabling “trust to travel” across organizations for smoother customer experiences. (Launch partners include Checkr (background data), Nova Credit (credit bureau), and others, whose services are now accessible through ProveX). Oct 2025 – Verified Agent for AI Commerce: In late 2025, Prove launched Prove Verified Agent, a solution to secure the emerging “agentic commerce” economy where AI agents act on behalf of users online. This initiative addresses the trust gap when autonomous AI bots make transactions. Prove’s Verified Agent creates a cryptographically verifiable chain-of-trust linking a human’s verified identity to their AI agent’s actions. It binds identity, intent, payment credentials, and user consent into an auditable unit for every AI-driven transaction. Essentially, an AI assistant gets its own Prove-verified identity token tied back to a real user, so counterparties can confirm “is this agent authorized and tied to a real person?” in real time. This system uses Prove’s Identity Graph as the backbone and replaces weak checks (like simple API keys) with strong multi-factor verification and cryptographic proof at each step. Use case: Imagine an AI shopping bot that orders on your behalf – Verified Agent ensures that bot presents a verifiable credential proving it’s acting for you, preventing fraudsters or rogue AIs from impersonating real customers. This is a forward-looking ML/security initiative to enable safe AI-to-business interactions. Oct 2025 – Global Identity Graph Platform: Prove rolled out its Global Identity Graph in October 2025, a foundational ML-powered identity platform covering over 10 billion devices and 90% of digital consumers worldwide. This graph continuously learns and updates reputational “trust signals” for each identity. For example, it tracks how long a phone number or device has been associated with a person and their history of trustworthy behavior vs. fraud signals. Unlike traditional identity data matching, Prove’s graph relies on deterministic ML models and real-time telecom data rather than static databases. A core use case is instant cross-border identity verification – e.g. a bank can instantly recognize a returning customer even if they changed phone carriers or moved countries, because the graph’s ML system has linked their various identifiers under one Prove ID token and noted continuous device tenure and behavior. The Global Identity Graph also acts like a “fraud antivirus” – providing real-time alerts if an identity’s pattern deviates (suggesting account takeover, SIM swap, or synthetic identity creation). This ML-driven platform was announced at Prove’s Improve 2025 conference and is now integrated into all Prove solutions globally. Aug 2025 – Unified Authentication (Passive MFA): Prove launched a new Unified Authentication solution that uses ML to passively recognize users via their phone and behavior, aiming to replace one-time passwords. This high-stakes production use case uses continuous risk scoring in the background: for each login or transaction, Prove’s model evaluates device ownership, SIM changes, location, etc., to silently authenticate legitimate users and flag suspicious activity. The company reported this approach reduced reliance on SMS OTPs and mitigated SIM swap fraud for clients. Essentially, the ML model identifies if a user’s phone session is low-risk (same trusted device, no anomalous changes) so they aren’t challenged, whereas if risk signals are high (e.g. recent SIM card change or impossible travel), it can prompt additional verification or block the action. This was promoted with ROI claims of cutting costs and abandonment rates, highlighting ML’s role in improving both security and user experience in customer authentication. June 2025 – Deepfake-Resistant Identity Verification: In mid-2025, Prove addressed the growing threat of AI-generated identity fraud (deepfakes, synthetic video) by partnering with authID on a “deepfake-resistant” verification capability. Through this partnership, Prove can augment its phone-centric verification with facial liveness detection and AI analysis to catch deepfake attempts. For example, during high-risk user onboarding, Prove’s workflow could invoke authID’s AI to verify that a selfie or video is of a real live person and not an AI-generated fake, before binding that face to a phone identity. This initiative shows Prove extending its ML toolkit by consuming a specialized computer-vision AI service to defend against novel fraud vectors (like generative AI spoofing). The partnership was announced as a way to “defend against AI‑generated fraud” globally. It underscores that AI/ML is in production use for fraud prevention – not just via Prove’s own models but also via integrated third-party AI where appropriate (in this case, to handle biometric verification, which is high-stakes for security). May 2025 – ML Platform Milestone (400M Transactions): By May 2025, Prove announced its Pre-Fill identity service had processed 400 million transactions and 150 million unique users, reflecting massive growth of its ML-driven platform. In the same announcement, Prove hinted at a “revolutionary advancement in frictionless identity authentication,” reinforcing that continuous innovation in ML was a “market mandate” to keep ahead of fraud. (While details were light, this suggests that around this time Prove was doubling down on enhancing its ML models and platform – which likely paved the way for the Global Identity Graph and other advancements launched later in 2025.) The scale here – hundreds of millions of verifications – demonstrates that Prove’s AI/ML systems (like Trust Score and Pre-Fill algorithms) have been operating at very high volume in production. Ongoing – Trust Score® Real-Time Risk Scoring: (Core ML use case, in production for several years)* Prove’s Trust Score® is an ML-powered risk score that evaluates the reputation of a phone number and device in real time. This is existential to Prove’s identity verification: every time a user signs up or logs in via a phone, Prove’s back-end computes a Trust Score (0–1000) based on myriad signals – e.g. SIM tenure, device change velocity, line type (mobile vs VoIP), number porting history, and recent SIM swap or account activity events. High scores indicate a “trusted” phone identity, whereas low scores flag risk of fraud. Concrete workload: A common deployment is before sending an OTP SMS – businesses call the Trust Score API to decide if the number is safe or if an OTP should be avoided. For instance, if the ML model returns a low score because it detects the phone is a virtual VoIP number or was SIM-swapped yesterday, the app can require additional ID checks instead of relying on a potentially compromised OTP. This use case represents streaming/real-time inference at scale: Prove reportedly measures 6 million+ “lifecycle events” daily and billions of authentications annually feeding into Trust Score computations. The Trust Score model is actively used in production by major banks, fintechs, and gaming platforms to prevent fraud like account takeovers and synthetic identities while letting legitimate users through with less friction. (Notably, Prove cites that using Trust Score as a pre-OTP check can significantly reduce OTP fraud; and scores above a certain threshold can auto-approve ~95% of users without extra steps.) This long-standing ML capability underpins many of Prove’s newer solutions and is continuously refined with new data.

import multiprocessing
import boto3
import subprocess
import os
import time

# Setup S3 client
s3_client = boto3.client('s3')

# Function to process data
def process_data(data):
    try:
        # Simulate processing
        result = subprocess.run(['python', 'process_script.py', data], capture_output=True)
        if result.returncode != 0:
            raise Exception('Processing failed')
        return result.stdout
    except Exception as e:
        print(f'Error processing {data}: {e}')
        return None

# Manual job queue and multiprocessing
def main():
    data_files = os.listdir('/data')
    pool = multiprocessing.Pool(4)
    results = []
    for data in data_files:
        result = pool.apply_async(process_data, args=(data,))
        results.append(result)
    pool.close()
    pool.join()

    # Save results to S3
    for result in results:
        output = result.get()
        if output:
            s3_client.put_object(Bucket='prove-bucket', Key=f'results/{output}.txt', Body=output)

if __name__ == '__main__':
    main()

# AFTER: Simplified batch processing using Ray on Anyscale for Prove
import ray

# Initialize Ray
ray.init()

# Define a Ray remote function
@ray.remote
def process_data(data):
    # Simulate processing
    return f'Processed {data}'

# Main function to distribute tasks
def main():
    data_files = ['/data/file1', '/data/file2', '/data/file3']
    # Distribute tasks across workers
    results = ray.get([process_data.remote(data) for data in data_files])
    
    # Auto-checkpoints, no manual S3 writes
    for result in results:
        print(f'Result: {result}')

if __name__ == '__main__':
    main()
    ray.shutdown()

# Prove's hyperparameter tuning with Ray Tune on Anyscale
import ray
from ray import tune
from my_ml_model import train_model

# Define search space
search_space = {
    'lr': tune.loguniform(1e-4, 1e-1),
    'batch_size': tune.choice([32, 64, 128])
}

# Use ASHA scheduler for efficient exploration
asha_scheduler = tune.schedulers.ASHA(max_t=100, grace_period=10, reduction_factor=2)

# Run tuning job with Ray Tune
ray.init()
tune.run(
    train_model,
    config=search_space,
    scheduler=asha_scheduler,
    num_samples=10,
    resources_per_trial={'cpu': 2, 'gpu': 1},  # auto-scaled resources
    local_dir='./ray_results',  # auto-checkpoints, no manual S3 writes
)
